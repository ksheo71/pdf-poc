# PDF 뷰어 구동 요약

PDF 파일이 화면에 보이기까지의 전체 과정을 단계별로 아주 자세하게 정리한 문서.

---

## 전체 구성 한 줄 요약
**원본 PDF 텍스트를 서버에서 지우고(redaction) PNG로 변환한 뒤 → 그 PNG를 브라우저에 표시하고 → 지워진 자리에 HTML `<div>` 오버레이를 덮어 "편집 가능한 PDF"처럼 보이게 하는 구조**이다.

---

## 1단계: 서버 부팅 — 모듈 로드 직후 (가장 중요한 단계)

[backend/app.py:191](backend/app.py#L191) 에서 **앱이 처음 import 되는 순간** 한 번만 실행된다.

```python
_png_cache, _meta_cache, _font_cache = _load_page1()
```

즉, 요청이 오기 전에 이미 모든 무거운 작업이 끝나 있다. `_load_page1()` 내부에서 일어나는 일:

### 1-1. PDF 열기 ([app.py:103-105](backend/app.py#L103-L105))
- `fitz.open(PDF_PATH)` — PyMuPDF로 `구성도.pdf` 열기
- `doc.load_page(0)` — 1페이지 로드
- `fitz.Matrix(SCALE, SCALE)` — 2배 확대 변환 행렬 준비 (고해상도 렌더링용)

### 1-2. 임베디드 폰트 추출 & 캐싱 ([app.py:107-117](backend/app.py#L107-L117))
- `doc.get_page_fonts(0)` 로 페이지에 쓰인 모든 폰트 목록 획득
- 각 폰트의 원본 바이너리(`buffer`)를 `font_cache[basefont]` 에 저장
- **이유**: 나중에 사용자가 편집한 텍스트를 PDF에 다시 쓸 때 "원본과 같은 폰트"를 그대로 재사용하기 위함

### 1-3. 프로브(probe) 픽셀맵 생성 ([app.py:119](backend/app.py#L119))
- `page.get_pixmap(matrix=matrix, alpha=False)` — PDF를 2배율 비트맵으로 미리 한 번 렌더링
- **용도**: 각 텍스트 박스 주변 픽셀 색을 샘플링해서 "배경색"을 알아내기 위함

### 1-4. 단어 추출 ([app.py:49-89, _extract_words_by_gap](backend/app.py#L49-L89))
- `page.get_text("rawdict")` — 문자 단위(character-level) 좌표를 모두 얻음
- 같은 span 안이라도 **문자 간 가로 간격이 폰트 크기의 0.3배를 넘으면 분리**
  - 이유: 표의 인접 셀("매출액|100" 같은 것)이 한 단어로 합쳐지지 않게 하기 위함
- 결과: `(x0, y0, x1, y1, text, span, origin_y)` 튜플 리스트

### 1-5. 블록 메타데이터 구성 ([app.py:121-147](backend/app.py#L121-L147))
각 단어마다:
- 좌표(`x0~y1`), 텍스트, 폰트명, 폰트 크기, 텍스트 색상(hex) 저장
- 숫자/기호만 있는 토큰은 폰트명 카운트에서 제외 → 문서에서 **가장 많이 쓰인 "문자용" 폰트**를 `dominant_letter_font`로 결정
- 왜? 숫자는 별도 폰트(예: Arial)로 태깅되어 있지만 시각적으로는 본문 폰트와 통일된 경우가 많아서, 나중에 텍스트 재작성 시 일관성 유지 목적

### 1-6. 핵심 — 텍스트 지우기 (Redaction) ([app.py:149-165](backend/app.py#L149-L165))
**이 단계가 "편집 가능한 PDF 뷰어"의 마술이다.**

각 텍스트 박스마다:
1. `_sample_bg()` — 박스 **바깥 4px 띠(상하좌우)** 의 픽셀 RGB 중앙값을 샘플링 → 주변 배경색 추정 ([app.py:20-42](backend/app.py#L20-L42))
2. `page.add_redact_annot(rect, fill=bg_color)` — 그 영역을 배경색으로 "덮는" redaction annotation 추가
3. 모든 블록 수집 후 `page.apply_redactions()` 한 번에 적용 → **PDF 페이지에서 모든 텍스트가 "증발"하고 그 자리에 주변색 사각형만 남음**

### 1-7. PNG 렌더링 & 메모리 캐싱 ([app.py:167-178](backend/app.py#L167-L178))
- 텍스트가 지워진 페이지를 `get_pixmap(matrix, alpha=False)` 로 2배율 렌더링
- `pix.tobytes("png")` → PNG 바이트열
- 메타데이터(`image_width`, `image_height`, `scale`, `blocks`) 구성
- 반환되어 모듈 전역 `_png_cache`, `_meta_cache`, `_font_cache` 에 저장

> ⚠️ 이 시점에서 이미 **이미지와 텍스트 좌표 목록이 메모리에 준비** 완료. 이후 요청은 모두 캐시 응답이라 즉시 끝난다.

## 2단계: 브라우저가 `/` 에 접속

[app.py:304](backend/app.py#L304) 의 `app.mount("/", StaticFiles(..., html=True))` 때문에 `/` 요청은 `frontend/index.html` 을 그대로 돌려준다.

브라우저가 HTML을 파싱하면서 **두 개의 리소스 요청**을 발행:
1. Tailwind CDN (`cdn.tailwindcss.com`) — 스타일링
2. `<img src="/api/page1/image.png">` [(index.html:78)](frontend/index.html#L78) — 텍스트가 지워진 PNG

## 3단계: 이미지 요청 → `/api/page1/image.png`

[app.py:219-221](backend/app.py#L219-L221):
```python
@app.get("/api/page1/image.png")
def get_page1_image():
    return Response(content=_png_cache, media_type="image/png")
```
캐시에 이미 PNG 바이트가 있으므로 **디스크/PDF 접근 없이 즉시 응답**. 브라우저가 이 PNG를 `<img>` 태그에 디코딩해서 그린다. 이 시점에서 사용자 눈에는 **"텍스트가 지워진 빈 구성도"**가 먼저 보인다.

## 4단계: 스크립트 실행 → 메타데이터 요청과 이미지 로드 대기 병렬 처리

[index.html:85-97](frontend/index.html#L85-L97):
```js
const [meta] = await Promise.all([
  fetch("/api/page1").then((r) => r.json()),
  new Promise((resolve) => {
    if (img.complete && img.naturalWidth > 0) resolve();
    else img.addEventListener("load", resolve, { once: true });
  }),
]);
```
- `/api/page1` 로 JSON 메타(`blocks[]` 포함) 요청
- 동시에 `<img>` 의 `load` 이벤트 대기
- **둘 다 끝나야** 다음 단계 진행 (이미지가 렌더된 크기를 알아야 오버레이 좌표를 계산할 수 있기 때문)

### 4-1. 서버 응답 [(app.py:214-216)](backend/app.py#L214-L216)
```python
@app.get("/api/page1")
def get_page1_meta():
    return _meta_cache
```
이것도 메모리 캐시 반환. `blocks[]` 에는 1단계에서 추출한 모든 텍스트의 **PDF 좌표계 기준 위치 + 텍스트 + 폰트 정보**가 들어 있다.

## 5단계: 오버레이 `<div>` 생성 ([index.html:100-139](frontend/index.html#L100-L139))

각 `block` 을 돌며:
1. `<div class="overlay" data-state="idle">` 생성 — **점선 파란 테두리** 상태
2. 내부에 4방향 `.joint` 원형 핸들(위/아래/좌/우) 4개 추가 — 평소엔 `display: none`, 선택 시 보임
3. 내부에 `<div class="editor" contentEditable="false">` 추가 — **원본 텍스트를 그대로 입력**, 색상도 원본 hex 적용
4. `viewer` 안에 append
5. 각 박스는 다음 두 좌표계를 보관:
   - `origRect` — 이미지 픽셀 좌표 (2배율 적용, **서버에 "이게 원본 위치였다"고 알려줄 때 사용**)
   - `imgRect` — 현재 위치 (사용자가 드래그/화살표로 이동하면 이 값만 바뀜)

---

## 6단계: 좌표 변환 — `layoutBox()` ([index.html:141-165](frontend/index.html#L141-L165))

여기서 "겹치게" 만드는 핵심 계산:
```js
const ratioX = img.clientWidth  / image_width;   // 브라우저 실제 표시폭 / 원본 PNG 폭
const ratioY = img.clientHeight / image_height;
left  = x0 * ratioX;
top   = y0 * ratioY;
width = (x1 - x0) * ratioX;
height= (y1 - y0) * ratioY;
```
- `<img>` 는 Tailwind의 `w-full h-auto` 로 화면 크기에 따라 **반응형 축소**되므로, 실제 렌더 크기 기준 비율을 매번 계산
- 이 값으로 오버레이의 `left/top/width/height` 를 설정 → **이미지에서 텍스트가 지워진 바로 그 자리에 `<div>` 가 정확히 겹침**
- 폰트 크기도 `box.fontSize * scale * ratioY` 로 맞춰서 편집 시 원본 글자 크기와 일치

### 6-1. 반응형 대응
```js
new ResizeObserver(layoutAll).observe(img);
```
창 크기가 바뀌어 이미지가 리사이즈되면 모든 오버레이를 **자동 재배치** → 항상 제자리.

## 7단계: 사용자에게 최종적으로 보이는 것

이 시점에 화면 구성은:

| 레이어 | 내용 |
|---|---|
| 맨 아래 | 텍스트가 지워진 `구성도.pdf` PNG (배경색으로 채워진 사각형만 남음) |
| 그 위 | 원본 텍스트를 담은 `<div class="editor">` 들이 **점선 파란 테두리** 오버레이로 각각 표시 |
| 최상단 | 상호작용 가능한 joint 핸들 (선택 시 노출) |

사용자 눈에는 **"원본 PDF가 그대로 렌더된 것처럼"** 보이지만, 실제로는 "지워진 이미지 + HTML 텍스트 오버레이"의 합성이다. 그래서 클릭해서 텍스트 수정, 위치 이동, 크기 조절이 가능한 것이다.

## 8단계(선택): 편집 → 다운로드 경로 요약

사용자가 편집하고 "PDF 다운로드" 버튼을 누르면 [(index.html:290-330)](frontend/index.html#L290-L330):
1. 변경된 박스만(`dirty` 필터) 추려서 `POST /api/commit` 으로 전송
2. 서버는 **원본 PDF를 다시 열고** ([app.py:228-245](backend/app.py#L228-L245)):
   - 이번엔 16배율(`COMMIT_SCALE`)로 정밀 배경색 샘플링
   - 변경된 위치를 redaction으로 지우고
   - 1단계에서 캐싱한 **원본 폰트 버퍼**를 `fitz.Font` 로 복원
   - `TextWriter` 로 편집된 텍스트를 새 위치에 동일 폰트/크기/색상으로 기록 ([app.py:255-288](backend/app.py#L255-L288))
3. 수정된 PDF 바이트 반환 → 프론트가 Blob으로 받아 `a.click()` 으로 다운로드 트리거

## 왜 이렇게 설계했나 — 설계 포인트 3가지

1. **무거운 작업은 부팅 시 1회만** — PDF 파싱/렌더/redaction은 수백 ms~수 초 걸리는 작업이므로 매 요청마다 하지 않고 전역 변수 캐시에 보관. 덕분에 사용자 체감은 정적 파일 서빙 수준.
2. **텍스트를 "지우고 다시 그리는"** 이유 — 브라우저에서 임베디드 PDF 폰트를 정확히 재현하기 어려우므로, PNG에서 텍스트를 완전히 제거하고 그 자리에 HTML 텍스트를 덮어 **편집 UX를 웹 네이티브로** 구현.
3. **두 좌표계 유지 (`origRect` vs `imgRect`)** — 다운로드 시 서버가 "어디를 지우고(원본 위치) / 어디에 새로 쓸지(현재 위치)"를 모두 알아야 하므로 프론트가 둘 다 보관.
